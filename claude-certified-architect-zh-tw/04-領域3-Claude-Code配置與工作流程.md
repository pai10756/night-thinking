# 領域 3：Claude Code 配置與工作流程（佔 20%）

> 這個領域的核心問題是：如何設定和客製化 Claude Code，讓團隊高效協作？

---

## 任務 3.1：配置 CLAUDE.md 檔案——層級結構、作用範圍與模組化

### 你需要理解的概念

**CLAUDE.md 是什麼？**

> 就是一個「給 AI 看的說明書」。你把專案的規則、慣例、注意事項寫在裡面，Claude Code 每次啟動都會讀它。

**三層配置層級**

```
優先順序（由上到下，越下面的越優先）：

1. 使用者級 ~/.claude/CLAUDE.md
   → 只對你自己有效，不會分享給隊友
   → 例如：你個人的偏好設定

2. 專案級 .claude/CLAUDE.md 或根目錄 CLAUDE.md
   → 整個團隊共用，透過 Git 分享
   → 例如：專案的程式碼風格、測試慣例

3. 目錄級 子目錄中的 CLAUDE.md
   → 只在編輯該目錄中的檔案時生效
   → 例如：前端元件的特殊規則
```

> **白話比喻：** 就像公司規則 → 部門規則 → 小組規則。越細的規則在適用範圍內越優先。

**常見問題：新成員沒收到指令**

如果有人把指令寫在自己的 `~/.claude/CLAUDE.md`（使用者級），新加入的團隊成員是看不到的。應該放在專案級配置中才能分享給所有人。

**`@import` 語法**

可以在 CLAUDE.md 中引用外部檔案，保持主檔案簡潔：
```
@import ./standards/api-conventions.md
@import ./standards/testing-rules.md
```

**`.claude/rules/` 資料夾**

把規則拆成多個獨立檔案，避免一個超長的 CLAUDE.md：
- `testing.md` — 測試相關規則
- `api-conventions.md` — API 慣例
- `deployment.md` — 部署規則

### 你需要具備的技能

- 診斷配置層級問題（例如：為什麼新人沒收到規則？因為放在使用者級而非專案級）
- 用 `@import` 針對每個套件引入相關的標準檔案
- 把大型 CLAUDE.md 拆成 `.claude/rules/` 中的小檔案
- 用 `/memory` 指令檢查哪些記憶檔案已載入，診斷不一致行為

## 任務 3.2：建立和配置自訂斜線命令與技能（Skills）

### 你需要理解的概念

**斜線命令（Slash Commands）**

就是在 Claude Code 中輸入 `/命令名` 就能執行的快捷指令。

| 類型 | 位置 | 分享方式 |
|------|------|---------|
| **專案範圍** | `.claude/commands/` | 透過 Git 分享，團隊都能用 |
| **個人範圍** | `~/.claude/commands/` | 只有自己能用 |

**技能（Skills）**

放在 `.claude/skills/` 中，用 `SKILL.md` 檔案定義。比斜線命令更強大，支援特殊配置：

| 前置設定（Frontmatter） | 說明 |
|----------------------|------|
| `context: fork` | 在隔離的環境中執行技能，避免技能的輸出「汙染」主對話 |
| `allowed-tools` | 限制技能執行時能使用的工具（例如只允許讀取，不允許刪除） |
| `argument-hint` | 當使用者沒提供參數時，提示需要輸入什麼 |

> **什麼是 `context: fork`？**
> 想像你在辦公室的白板上畫了很多草圖做腦力激盪。如果用 `context: fork`，這些草圖會畫在一張「臨時紙」上，做完後只把結論帶回來，不會弄亂你的白板。

**技能 vs CLAUDE.md 的選擇**

| | 技能（Skills） | CLAUDE.md |
|--|--------------|-----------|
| 載入時機 | 手動觸發（按需使用） | 每次都自動載入 |
| 適用場景 | 特定任務的工作流程 | 通用的專案規則和標準 |

### 你需要具備的技能

- 在 `.claude/commands/` 建立專案範圍的斜線命令
- 用 `context: fork` 隔離會產生大量輸出的技能
- 用 `allowed-tools` 限制技能的工具存取權限
- 用 `argument-hint` 提示使用者輸入必要參數
- 根據場景選擇用技能還是 CLAUDE.md

## 任務 3.3：用路徑規則實現條件式的慣例載入

### 你需要理解的概念

**路徑規則（Path-Specific Rules）**

在 `.claude/rules/` 中建立規則檔案，用 YAML 前置資料（frontmatter）指定「只在編輯特定路徑的檔案時才載入」：

```yaml
---
paths: ["src/api/**/*"]
---
# API 開發慣例
- 所有回應必須包含 status 欄位
- 錯誤回應必須包含 error_code 和 message
```

> **什麼是 Glob 模式？**
> 一種用萬用字元來配對檔名的寫法：
> - `*` = 任何檔名
> - `**` = 任何目錄層級
> - `*.test.tsx` = 所有以 `.test.tsx` 結尾的檔案
> - `src/api/**/*` = src/api 目錄下所有檔案（含子目錄）

**為什麼比目錄級 CLAUDE.md 更好？**

測試檔案通常散落在各處（`Button.test.tsx` 放在 `Button.tsx` 旁邊），你不可能在每個目錄都放一份 CLAUDE.md。但用路徑規則 `paths: ["**/*.test.tsx"]` 就能一次覆蓋所有測試檔案。

### 你需要具備的技能

- 建立帶有路徑 glob 模式的規則檔案
- 用 glob 模式按檔案類型（而非目錄位置）套用規則
- 判斷何時該用路徑規則（跨目錄的慣例）vs 目錄級 CLAUDE.md（特定目錄的慣例）

## 任務 3.4：判斷何時該用計畫模式 vs 直接執行

### 你需要理解的概念

| | 計畫模式（Plan Mode） | 直接執行（Direct Execution） |
|--|-------------------|--------------------------|
| **適用場景** | 複雜任務、大規模變更、多種做法可選、需要架構決策 | 簡單、範圍明確的變更 |
| **例子** | 微服務重構、影響 45+ 檔案的程式庫遷移 | 修一個單檔 Bug、加一行驗證邏輯 |
| **優點** | 先探索再動手，避免代價高昂的返工 | 快速完成，不浪費時間 |

> **白話比喻：**
> - 計畫模式 = 先看地圖再開車（適合長途旅行）
> - 直接執行 = 直接出發（適合去巷口便利商店）

**Explore 子代理人**

探索程式碼庫時會產生大量輸出。用 Explore 子代理人在隔離環境中探索，只把摘要帶回主對話，避免主對話的上下文被塞滿。

### 你需要具備的技能

- 需要架構決策時選計畫模式
- 範圍明確的小改動選直接執行
- 用 Explore 子代理人做冗長的探索
- 混合使用：計畫模式做調查 → 直接執行做實作

## 任務 3.5：運用反覆改進技巧逐步提升品質

### 你需要理解的概念

**四種改進技巧：**

1. **具體的輸入/輸出範例**
   - 當文字描述被 AI 理解得不一致時，直接給 2-3 個「輸入→輸出」的實例最有效
   - 例如：「輸入：`2024-01-15` → 輸出：`民國113年1月15日`」

2. **測試驅動的反覆改進**
   - 先寫好測試（包含正常情況、邊界情況、效能要求）
   - 讓 AI 寫程式碼
   - 把測試失敗的結果回饋給 AI，引導它逐步修正

3. **訪談模式（Interview Pattern）**
   - 讓 Claude 在動手之前先問你問題
   - 幫你發現你自己可能沒想到的考量（例如快取失效策略、失敗模式等）

4. **單次 vs 分次回饋**
   - 如果問題彼此相關（修一個會影響另一個）→ 一次說完
   - 如果問題各自獨立 → 一次修一個

### 你需要具備的技能

- 提供具體的輸入/輸出範例來澄清模糊的需求
- 先寫測試再開發，用測試失敗引導改進
- 在不熟悉的領域使用訪談模式
- 判斷何時一次給所有回饋 vs 逐步修正

## 任務 3.6：將 Claude Code 整合到 CI/CD 流程

### 你需要理解的概念

**`-p` 旗標（非互動模式）**

在自動化流程中執行 Claude Code 時，必須加上 `-p`（或 `--print`）旗標。否則 Claude Code 會等待人類輸入，導致流程卡住。

```bash
# ❌ 會卡住
claude "分析這個 PR 的安全問題"

# ✅ 正確
claude -p "分析這個 PR 的安全問題"
```

**結構化輸出**

在 CI 環境中，用 `--output-format json` 和 `--json-schema` 來確保輸出是機器可解析的 JSON 格式，方便自動化處理（例如自動發佈 PR 評論）。

**上下文隔離**

> 同一個 Claude 工作階段既產生程式碼又審查程式碼，效果不好。因為它會「記得」自己的推理過程，不太容易質疑自己的決定。
>
> 就像作家自己校對自己的文章，很容易看不到自己的錯誤。應該用一個獨立的 Claude 實例來做審查。

**CLAUDE.md 在 CI 中的作用**

把測試標準、可用的測試資料、審查標準等寫在 CLAUDE.md 中，讓 CI 中的 Claude Code 也能讀到這些規則。

### 你需要具備的技能

- 在 CI 中用 `-p` 旗標執行 Claude Code
- 用 `--output-format json` + `--json-schema` 產出結構化結果
- 重新審查時把之前的發現放在上下文中，讓 Claude 只回報新的或未解決的問題
- 提供現有測試檔案以避免產生重複的測試
- 在 CLAUDE.md 中記錄測試標準和可用資源
