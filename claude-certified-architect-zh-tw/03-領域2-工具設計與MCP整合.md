# 領域 2：工具設計與 MCP 整合（佔 18%）

> 這個領域的核心問題是：如何設計好用的工具讓 AI 正確選擇和使用？如何用 MCP 連接外部服務？

---

## 任務 2.1：設計有效的工具介面——清楚的描述與明確的邊界

### 你需要理解的概念

**工具描述是 AI 選擇工具的主要依據**

當 AI 面前有多個工具可以選擇時，它是靠「工具描述」來判斷該用哪一個。如果描述寫得太簡略，AI 就容易選錯。

> **白話比喻：** 想像你打開一個工具箱，裡面每個工具都只貼了一張小紙條。如果紙條只寫「工具 A」「工具 B」，你不知道哪個是螺絲起子、哪個是扳手。但如果寫了「十字螺絲起子——適用 M3-M5 螺絲」，你就能精確挑選。

**常見問題：工具描述太相似**

例如：
- `analyze_content`：「分析內容」
- `analyze_document`：「分析文件」

這兩個描述幾乎一樣，AI 會搞混。解法是重新命名並詳細描述：
- `extract_web_results`：「從網頁搜尋結果中擷取關鍵資訊。輸入：搜尋結果 JSON。輸出：摘要清單。」
- `summarize_pdf_document`：「摘要 PDF 文件的主要內容。輸入：PDF 文件路徑。輸出：結構化摘要。」

**系統提示詞中的關鍵字也會影響工具選擇**

如果系統提示詞中寫「遇到文件相關問題就用 analyze」，可能會讓 AI 不管三七二十一都去呼叫名字有 analyze 的工具。

### 你需要具備的技能

- 寫清楚每個工具的用途、預期輸入、輸出、以及「什麼時候該用它而非類似工具」
- 重新命名和更新描述以消除功能重疊
- 把一個大而全的工具拆成多個專用工具（例如把 `analyze_document` 拆成 `extract_data_points`、`summarize_content`、`verify_claim_against_source`）
- 檢查系統提示詞是否有可能誤導 AI 的關鍵字

## 任務 2.2：為 MCP 工具實作結構化的錯誤回應

### 你需要理解的概念

**錯誤不是都一樣的！**

| 錯誤類型 | 說明 | 舉例 | 可以重試？ |
|---------|------|------|---------|
| **暫時性錯誤（Transient）** | 暫時的問題，等一下可能就好了 | 網路逾時、服務暫時不可用 | ✅ 可以 |
| **驗證錯誤（Validation）** | 輸入的資料有問題 | 格式錯誤、缺少必填欄位 | ❌ 除非修正輸入 |
| **商業邏輯錯誤（Business）** | 違反公司政策 | 退款超過上限、優惠券已過期 | ❌ 不可以 |
| **權限錯誤（Permission）** | 沒有執行這個操作的權限 | 一般客服不能批准大額退款 | ❌ 不可以 |

**為什麼「操作失敗」這種籠統錯誤訊息不好？**

如果所有錯誤都只回傳「操作失敗」，AI 就無法判斷應該重試、換方式處理、還是轉給人類。

> **白話比喻：** 就像去醫院看病，如果醫生只說「你不舒服」但不告訴你是感冒還是骨折，你就不知道該吃藥還是打石膏。

**MCP 的 `isError` 旗標**

MCP 工具回傳時可以設定 `isError` 標記，告訴 AI 這次工具呼叫失敗了。

### 你需要具備的技能

- 回傳結構化錯誤資訊，包含：
  - `errorCategory`（錯誤類別）：transient / validation / permission
  - `isRetryable`（可否重試）：true / false
  - 人類可讀的描述
- 對商業邏輯錯誤，加上 `retriable: false` 和客戶友善的解釋
- 子代理人自行處理暫時性錯誤（重試），只把無法解決的錯誤上報給協調者
- 區分「存取失敗」（需要重試）和「有效的空結果」（查詢成功但確實沒有資料）

> **白話比喻：** 「存取失敗」就像打電話沒人接（可能對方在忙，等一下再打）；「有效的空結果」就像打電話問「有沒有這個人？」得到回答「沒有」——查詢本身是成功的。

## 任務 2.3：合理分配工具給不同代理人，並配置工具選擇

### 你需要理解的概念

**工具太多反而會出問題**

給一個 AI 代理人 18 個工具，它的選擇可靠度會比只給 4-5 個工具低得多。工具越多，決策越複雜，出錯機率越高。

> **白話比喻：** 就像餐廳菜單有 200 道菜，你反而不知道點什麼。但如果只有 5 道精選，你很快就能選好。

**不要給代理人它不需要的工具**

例如，一個專門做「綜合整理」的代理人，如果你給它「網路搜尋」工具，它可能會忍不住自己去搜，而不是用之前搜尋代理人已經找好的資料。

**`tool_choice` 配置選項**

| 選項 | 效果 | 適用場景 |
|------|------|---------|
| `"auto"` | AI 自己決定要不要用工具、用哪個 | 一般情況 |
| `"any"` | AI 必須呼叫一個工具（不能只回文字） | 你需要保證得到結構化輸出 |
| `{"type": "tool", "name": "xxx"}` | 強制呼叫指定的工具 | 第一步一定要做某件事時 |

### 你需要具備的技能

- 限制每個子代理人只能用與其角色相關的工具
- 把通用工具替換成受限的版本（例如把 `fetch_url` 換成只能載入文件的 `load_document`）
- 為高頻需求提供「跨角色」的小工具（例如給綜合代理人一個 `verify_fact` 小工具做簡單查核）
- 用強制工具選擇確保第一步一定執行某工具
- 用 `tool_choice: "any"` 保證 AI 一定回傳工具呼叫而非純文字

## 任務 2.4：將 MCP 伺服器整合進 Claude Code 和代理人工作流程

### 你需要理解的概念

**MCP 伺服器的兩種範圍**

| 範圍 | 設定檔 | 說明 |
|------|--------|------|
| **專案級**（Project） | `.mcp.json`（放在專案目錄中） | 團隊共用，透過版本控制（如 Git）分享 |
| **使用者級**（User） | `~/.claude.json`（放在家目錄） | 個人使用，不影響團隊 |

> **什麼是版本控制？**
> 就像 Google Docs 的「版本歷史記錄」——每次修改都會被記錄下來，可以追溯和還原。Git 是最流行的版本控制工具。

**環境變數擴展**

在 `.mcp.json` 中可以用 `${GITHUB_TOKEN}` 這樣的寫法，系統會自動替換成實際的密碼或金鑰。這樣密碼就不會被直接寫在檔案裡（避免被上傳到公開的程式碼庫）。

**MCP 資源（Resources）**

> **工具（Tools）vs 資源（Resources）：**
> - **工具**：能「做事」的功能（例如搜尋、建立記錄、發送通知）
> - **資源**：能「提供資訊」的內容目錄（例如 Issue 清單、文件結構、資料庫欄位定義）
>
> 資源讓 AI 先看到「有什麼可用的」，減少盲目嘗試的次數。

### 你需要具備的技能

- 在 `.mcp.json` 配置團隊共用的 MCP 伺服器（搭配環境變數）
- 在 `~/.claude.json` 配置個人實驗用的 MCP 伺服器
- 強化 MCP 工具描述，防止 AI 偏好使用內建工具（如 Grep）而忽略功能更強的 MCP 工具
- 優先使用現有的社群 MCP 伺服器（如 Jira 整合），只在需要團隊特有功能時才自建
- 把內容目錄以 MCP 資源的形式公開，讓 AI 知道有什麼資料可用

## 任務 2.5：有效選用內建工具（Read、Write、Edit、Bash、Grep、Glob）

### 你需要理解的概念

| 工具 | 用途 | 什麼時候用？ |
|------|------|------------|
| **Grep** | 搜尋檔案**內容**（在程式碼中找特定文字） | 「找出所有呼叫 `processPayment` 的地方」 |
| **Glob** | 搜尋**檔名**模式（找特定類型的檔案） | 「找出所有 `.test.tsx` 結尾的測試檔案」 |
| **Read** | 讀取整個檔案的內容 | 「讓我看看這個檔案裡寫了什麼」 |
| **Write** | 寫入整個檔案 | 建立新檔案，或完全覆寫 |
| **Edit** | 精確修改檔案中的特定部分 | 「只改這個函式的第 3 行」 |
| **Bash** | 執行命令列指令 | 跑測試、安裝套件、執行腳本 |

> **Grep vs Glob 的差別：**
> - Grep = 找「內容」（在檔案裡面找特定文字）
> - Glob = 找「檔案」（用名稱模式找檔案）

**當 Edit 失敗時的備案**

Edit 工具需要找到「唯一匹配的文字」才能修改。如果要修改的文字在檔案中出現多次，Edit 會失敗。這時候可以先用 Read 讀取整個檔案，再用 Write 寫回修改後的版本。

### 你需要具備的技能

- 根據需求選擇正確的工具
- 漸進式理解程式碼庫：先用 Grep 找入口 → 再用 Read 追蹤引用 → 逐步建立全貌（而不是一開始就把所有檔案都讀一遍）
- 追蹤函式使用情況：先找出所有匯出的名稱，再逐一搜尋它們在程式碼庫中被使用的地方
