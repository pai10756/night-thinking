# 領域 1：AI 代理人架構與協調（佔 27%）

> 這是考試中佔比最高的領域。核心問題是：如何設計讓 AI 能自動完成多步驟任務的系統？

---

## 任務 1.1：設計和實作「代理人迴圈」以自動執行任務

### 你需要理解的概念

**代理人迴圈（Agentic Loop）是什麼？**

想像你請一位助理幫你處理一件事。助理的工作流程是：

```
你下指令 → 助理思考 → 助理使用工具 → 看結果 → 繼續或完成
    ↑                                        │
    └────────── 如果還沒做完就繼續 ──────────┘
```

這就是「代理人迴圈」——AI 不斷地：
1. 接收指令
2. 決定用哪個工具
3. 執行工具
4. 檢查結果
5. 判斷：要繼續做，還是已經完成了？

**關鍵概念：`stop_reason`（停止原因）**

每次 Claude 回應時，會附帶一個「停止原因」：
- `"tool_use"`（使用工具）：代表 Claude 想要呼叫一個工具。你應該執行工具，把結果送回去，讓迴圈繼續。
- `"end_turn"`（結束回合）：代表 Claude 認為任務完成了。迴圈應該結束。

> **白話比喻：** 就像餐廳點餐。服務生每次端一道菜來時都會問「還需要什麼嗎？」如果你說「來一杯水」（tool_use），他就繼續服務；如果你說「不用了，謝謝」（end_turn），服務就結束了。

**模型驅動的決策 vs 預設流程**
- **模型驅動**：Claude 根據當前情況自己判斷下一步該做什麼（推薦做法）
- **預設流程**：事先寫好固定的步驟順序，不管情況如何都照做（較不靈活）

### 你需要具備的技能

- 實作迴圈控制：當 `stop_reason` 是 `"tool_use"` 時繼續，是 `"end_turn"` 時結束
- 每次工具執行完畢後，把結果加進對話歷史，讓 Claude 能根據新資訊做判斷
- **避免常見錯誤做法（反模式）**：
  - ❌ 用解析 AI 回覆的文字來判斷是否該停止（不可靠）
  - ❌ 設定一個固定次數上限作為主要停止機制（例如「最多跑 5 次就停」）
  - ❌ 檢查 AI 是否回覆了文字內容來判斷是否完成

## 任務 1.2：用「協調者-子代理人」模式來管理多代理人系統

### 你需要理解的概念

**中心輻射架構（Hub-and-Spoke）**

```
        子代理人A（搜尋）
           ↑
           │
子代理人C ← 協調者 → 子代理人B（分析）
（報告）   │
           ↓
        子代理人D（驗證）
```

- **協調者（Coordinator）**：就像團隊主管，負責分配工作、整合結果、處理錯誤
- **子代理人（Subagent）**：就像團隊成員，各有專長，只負責自己的任務
- 所有溝通都透過協調者——子代理人之間不直接對話

**重要：子代理人是獨立的**
- 子代理人**不會自動繼承**協調者的對話歷史
- 你必須明確地把需要的資訊傳給每個子代理人

> **白話比喻：** 子代理人就像外包廠商。你（協調者）不能假設外包商知道你公司內部的背景資訊，你必須在工作說明書中寫清楚所有他需要知道的事。

**任務分解不能太狹窄**
- 如果協調者把「AI 對創意產業的影響」這個主題，只分解成「AI 在數位藝術」「AI 在攝影」「AI 在平面設計」，就會完全漏掉音樂、寫作、電影製作

### 你需要具備的技能

- 設計能動態選擇子代理人的協調者（不是每次都啟動所有子代理人）
- 分配研究範圍以避免重複（例如，讓不同子代理人負責不同子主題）
- 實作「反覆改進迴圈」：協調者檢查綜合結果 → 發現漏洞 → 重新指派搜尋和分析 → 重新綜合 → 直到覆蓋充分
- 所有子代理人的溝通都經過協調者（便於監控和統一處理錯誤）

## 任務 1.3：配置子代理人的啟動、上下文傳遞與產生

### 你需要理解的概念

- **Task 工具**：用來「生出」子代理人的機制。協調者的 `allowedTools`（允許使用的工具清單）必須包含 `"Task"` 才能啟動子代理人
- **上下文必須明確傳遞**：子代理人不會自動知道之前發生了什麼，你必須在啟動它的時候把需要的資訊寫在指令中
- **AgentDefinition（代理人定義）**：每個子代理人的「規格表」，包含描述、系統提示詞、允許使用的工具
- **分支會話（fork_session）**：從同一個分析起點，分出獨立的探索路線

> **白話比喻：** fork_session 就像在解決方案上做 A/B 測試——你在同一個起點分出兩條路，分別嘗試不同的做法，看哪個更好。

### 你需要具備的技能

- 把前面代理人的完整發現直接寫進下一個子代理人的指令中
- 用結構化格式分離「內容」和「元資料」（來源網址、文件名、頁碼）以保留出處
- **平行啟動多個子代理人**：在協調者的同一次回應中發出多個 Task 工具呼叫（而不是分成多個回合）
- 設計指令時描述「目標和品質標準」而非「一步步怎麼做」，讓子代理人有應變的彈性

## 任務 1.4：實作多步驟工作流程與交接模式

### 你需要理解的概念

**程式化強制 vs 提示詞引導**

| 方式 | 說明 | 可靠度 |
|------|------|--------|
| **程式化強制**（Hooks、先決條件閘門） | 用程式碼硬性規定：「沒做 A 就不能做 B」 | 100% 可靠 |
| **提示詞引導** | 在指令中告訴 AI「請先做 A 再做 B」 | 大部分時候有效，但有時會被忽略 |

> **什麼時候一定要用程式化強制？**
> 當後果嚴重時。例如：在金融操作前必須驗證身分。光靠提示詞說「請先驗證身分」有時 AI 會跳過這步——如果漏掉就可能退款到錯誤的帳戶。

**結構化交接協定**
當 AI 需要把案件轉交給真人時，應該提供一份完整的摘要：
- 客戶 ID
- 問題根本原因
- 退款金額
- 建議處理方式

> 因為真人客服看不到之前 AI 和客戶的完整對話記錄，所以這份摘要至關重要。

### 你需要具備的技能

- 實作「先決條件閘門」：例如，不讓 `process_refund`（處理退款）執行，除非 `get_customer`（查詢客戶）已經回傳了已驗證的客戶 ID
- 將多重問題的客戶請求拆解成獨立項目，平行處理後綜合出統一的解決方案
- 在升級給真人時，編寫結構化的交接摘要

## 任務 1.5：使用 Agent SDK Hooks 攔截工具呼叫與資料標準化

### 你需要理解的概念

**Hooks（鉤子）是什麼？**

> 想像你在門口裝了一個自動門鈴——有人經過時自動觸發。Hooks 就像這樣：在特定事件發生時，自動執行你設定的動作。

- **PostToolUse Hook**（工具使用後的鉤子）：工具回傳結果後，在 AI 看到之前，先做一些處理
  - 例如：不同工具回傳的日期格式不一樣（有的是 Unix 時間戳、有的是 ISO 8601），Hook 可以統一成同一種格式
- **工具呼叫攔截 Hook**：AI 要使用工具前，先檢查是否符合規則
  - 例如：退款金額超過 500 美元就擋下，轉到人工處理

### 你需要具備的技能

- 實作 PostToolUse Hook 來統一不同 MCP 工具回傳的資料格式
- 實作攔截 Hook 來阻止違反政策的操作，並轉到替代流程
- 當商業規則需要 100% 遵守時，選擇 Hook 而非提示詞

## 任務 1.6：設計複雜工作流程的任務分解策略

### 你需要理解的概念

**兩種任務分解模式：**

| 模式 | 說明 | 適用場景 |
|------|------|---------|
| **提示鏈（Prompt Chaining）** | 固定的順序步驟：先做 A → 再做 B → 最後做 C | 可預測的多面向審查 |
| **動態適應分解** | 根據每一步的發現，動態決定下一步做什麼 | 開放式的調查任務 |

> **白話比喻：**
> - 提示鏈像食譜：按步驟來，先切菜、再炒菜、最後裝盤
> - 動態分解像偵探辦案：每找到一個線索，就決定下一步去哪裡調查

### 你需要具備的技能

- 根據任務性質選擇合適的分解模式
- 大型程式碼審查：先逐檔分析，再做跨檔整合審查（避免「注意力稀釋」——一次看太多檔案，每個都只看到皮毛）
- 開放式任務（如「幫舊程式碼加上完整測試」）：先了解結構 → 找出高影響區域 → 建立優先順序計畫 → 根據發現調整計畫

## 任務 1.7：管理會話狀態、恢復與分支

### 你需要理解的概念

| 功能 | 說明 |
|------|------|
| `--resume <session-name>` | 用名字恢復之前的對話（像書籤一樣，下次打開就回到上次的地方） |
| `fork_session` | 從某個分析點分出獨立分支，用來探索不同方案 |
| 告知變更 | 恢復會話後，要告訴 AI 哪些檔案在你離開期間被改過了 |
| 重新開始 vs 恢復 | 如果舊會話的工具結果已經過時，用結構化摘要開始新會話比恢復舊會話更可靠 |

### 你需要具備的技能

- 用 `--resume` 繼續之前的調查
- 用 `fork_session` 建立平行探索（例如比較兩種測試策略）
- 判斷何時該恢復（舊資訊大部分還有效）vs 何時該重新開始（舊工具結果已過時）
- 恢復會話後，告知 AI 特定檔案的變更以做針對性重新分析
