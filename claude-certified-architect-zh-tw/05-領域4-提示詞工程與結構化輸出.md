# 領域 4：提示詞工程與結構化輸出（佔 20%）

> 這個領域的核心問題是：如何寫出精確的指令，讓 AI 產出格式正確、內容可靠的結果？

---

## 任務 4.1：用明確的標準設計提示詞，提高精確度、減少誤報

### 你需要理解的概念

**明確標準 vs 模糊指令**

| 寫法 | 效果 |
|------|------|
| ❌ 「檢查註解是否正確」 | AI 不知道什麼叫「正確」，可能把無害的風格差異也標記出來 |
| ✅ 「只在註解描述的行為與程式碼實際行為矛盾時才標記」 | AI 有明確的判斷標準 |

**「保守一點」這類籠統指令沒用**

告訴 AI「只報告高信心的發現」或「保守一點」，幾乎不會改善精確度。因為 AI 不知道你心中的「保守」門檻在哪裡。

> **白話比喻：** 就像告訴新員工「嚴格一點」——每個人對「嚴格」的定義不同。你應該直接說「退款超過 1000 元的才需要主管簽核」。

**誤報對信任的傷害**

如果 AI 的程式碼審查有太多誤報（說有問題但其實沒有），開發者會對所有建議都失去信心——包括真正有問題的那些。

### 你需要具備的技能

- 寫出具體的審查標準：明確定義哪些問題要報告（Bug、安全漏洞）、哪些要跳過（小風格問題）
- 暫時停用誤報率高的類別，在改善提示詞後再恢復
- 用具體的程式碼範例定義每個嚴重等級

---

## 任務 4.2：用 Few-shot 範例提升輸出一致性和品質

### 你需要理解的概念

**什麼是 Few-shot 範例？**

> 在提示詞中附上幾個「示範」，讓 AI 看到你期望的格式和判斷方式。
>
> 就像新員工到職第一天，你不是只給他一本厚厚的規則手冊，而是直接帶他看幾個「這樣做是對的、那樣做是錯的」的實例。

**Few-shot 的強大之處：**

1. **格式一致性**：當詳細指令還是產出不一致的格式時，2-4 個範例就能大幅改善
2. **模糊情況處理**：展示在邊界情況下的正確判斷
3. **泛化能力**：AI 不是只會照抄範例，而是學會背後的判斷邏輯，能舉一反三
4. **減少幻覺**：在資料擷取任務中，範例能教 AI 正確處理非標準的資料格式

### 你需要具備的技能

- 針對模糊場景建立 2-4 個 few-shot 範例，展示選擇的理由
- 範例中包含期望的輸出格式（位置、問題、嚴重度、建議修正）
- 提供區分「可接受的程式碼模式」和「真正問題」的範例
- 用範例展示如何處理不同結構的文件（行內引用 vs 參考文獻、章節結構 vs 散落式）
- 當某個欄位常常擷取失敗時，增加展示正確擷取方式的範例

## 任務 4.3：用 Tool Use 和 JSON Schema 強制輸出結構化資料

### 你需要理解的概念

**為什麼用 Tool Use 而非純文字？**

| 方式 | JSON 語法錯誤？ | 欄位遺漏？ | 格式錯誤？ |
|------|-------------|---------|---------|
| 純文字要求「請輸出 JSON」 | ⚠️ 有時候會 | ⚠️ 有時候會 | ⚠️ 有時候會 |
| **Tool Use + JSON Schema** | ✅ 不可能 | ✅ 不可能 | ✅ 不可能 |

> **什麼是 Tool Use？**
> 不是叫 AI「回覆一段 JSON 文字」，而是讓 AI「呼叫一個工具」，工具的輸入參數就是你定義好格式的 JSON Schema。這樣 AI 的輸出一定符合你定義的格式。

**但 Tool Use 無法防止「語意錯誤」**

格式一定是對的，但「值」可能是錯的。例如：
- 明細項目的金額加起來不等於總金額
- 資料放到了錯誤的欄位

> **白話比喻：** Tool Use 保證表格每一格都填了正確類型的資料（數字欄填數字、日期欄填日期），但無法保證填的「內容」是對的。

**`tool_choice` 三種模式複習：**

| 模式 | AI 的行為 | 適用場景 |
|------|---------|---------|
| `"auto"` | AI 自己決定用不用工具 | 一般對話 |
| `"any"` | AI 一定會呼叫某個工具 | 多個 Schema 但不確定文件類型 |
| `{"type":"tool","name":"xxx"}` | AI 一定呼叫指定的工具 | 強制第一步做特定擷取 |

**Schema 設計技巧：**

- **必填 vs 選填欄位**：如果原始文件可能不包含某項資訊，把該欄位設為**選填（nullable）**。否則 AI 會為了填滿必填欄位而「編造」資料。
- **Enum 加 "other"**：分類欄位用固定選項（enum），但加一個 `"other"` + 附加說明欄位，處理不在預期中的類別。
- **`"unclear"` 值**：對模糊的情況提供一個「不確定」的選項。

### 你需要具備的技能

- 定義擷取工具的 JSON Schema 並從 `tool_use` 回應中提取資料
- 用 `tool_choice: "any"` 保證得到結構化輸出
- 用強制工具選擇確保特定擷取先執行
- 把可能缺失的欄位設為 nullable
- 加上 `"unclear"` 和 `"other"` + 說明欄位
- 在提示詞中加上格式標準化規則來處理不一致的原始格式

## 任務 4.4：實作驗證、重試和回饋迴圈以提升擷取品質

### 你需要理解的概念

**重試加錯誤回饋（Retry with Error Feedback）**

```
第一次擷取 → 驗證失敗 → 把「原始文件 + 失敗的擷取結果 + 具體錯誤訊息」
                         一起送回給 AI → AI 自我修正 → 重新擷取
```

> **白話比喻：** 就像老師改作文。不是只說「重寫」，而是具體指出「第二段邏輯不通、第三段缺少例子」，學生才知道怎麼改。

**重試的極限：資訊不存在時重試沒用**

如果原始文件根本沒有你要的資訊（例如文件沒寫作者名字），重試再多次也擷取不出來。重試只對「格式錯誤」「結構誤解」有效。

**回饋迴圈設計**

在擷取結果中加入 `detected_pattern`（偵測到的模式）欄位。當開發者標記某個發現為「誤報」時，系統可以分析是哪種模式容易造成誤報，進而改善。

**自我驗證流程**

- 同時擷取「計算總額」和「文件標示的總額」，讓系統能自動發現差異
- 加入 `conflict_detected`（偵測到衝突）布林欄位，標記原始資料本身就矛盾的情況

### 你需要具備的技能

- 實作帶有具體錯誤訊息的重試流程
- 判斷何時重試有效（格式錯誤）vs 何時無效（資訊不存在）
- 在結構化發現中加入 `detected_pattern` 以分析誤報模式
- 設計自我驗證流程

## 任務 4.5：設計高效的批次處理策略

### 你需要理解的概念

**Message Batches API（訊息批次 API）**

| 特性 | 說明 |
|------|------|
| 成本節省 | **省 50%** |
| 處理時間 | 最長 24 小時（沒有保證的速度 SLA） |
| 適用場景 | 不急的工作：隔夜報告、每週稽核、夜間測試產生 |
| 不適用場景 | 急需結果的工作：合併前的程式碼檢查 |
| 限制 | 不支援多輪工具呼叫（不能在批次中間執行工具再回傳結果） |

> **什麼是 SLA？**
> Service Level Agreement（服務等級協議），就是服務商承諾的品質標準。「沒有 SLA」意味著沒有「保證幾小時內完成」的承諾。

**`custom_id` 欄位**

每個批次請求都有一個你自訂的 ID，用來在收到結果時對應「這個結果是哪個請求的」。

> **白話比喻：** 就像送洗衣服時，每件衣服都掛一個號碼牌，取衣時才知道哪件是誰的。

### 你需要具備的技能

- 根據時間需求選擇：急用選同步 API、不急選批次 API
- 計算批次提交頻率（例如：批次最多 24 小時，要保證 30 小時 SLA → 至少每 4 小時提交一次）
- 處理批次失敗：用 `custom_id` 找出失敗的文件，修改後（例如把太長的文件切小）重新提交
- 先在小樣本上調好提示詞，再批次處理大量文件（減少重試成本）

## 任務 4.6：設計多實例和多輪次審查架構

### 你需要理解的概念

**AI 自我審查的侷限**

> 同一個 AI 工作階段產生程式碼後再自己審查，效果很差。因為它還「記得」自己為什麼這樣寫，不太會質疑自己的決定。
>
> **白話比喻：** 你剛寫完一封信，馬上自己校對，通常看不到自己的錯字。但讓別人來看，一眼就發現了。

**獨立審查實例更有效**

用一個**全新的、沒有之前推理過程的** Claude 來審查，比叫同一個 Claude 「仔細想想有沒有問題」更能抓到細微的 Bug。

**多輪次審查（Multi-pass Review）**

大型程式碼審查（例如 14 個檔案）不要一次看完。分成：
1. **逐檔審查**：每個檔案各自分析（找出該檔案內的問題）
2. **跨檔整合審查**：分析檔案之間的資料流和互動

> 一次審查太多檔案會造成「注意力稀釋」——有些檔案被仔細看、有些只是草草帶過，甚至可能在不同檔案中對同樣的程式碼給出相反的評價。

### 你需要具備的技能

- 用第二個獨立的 Claude 實例來審查產出的程式碼
- 把大型審查拆成逐檔 + 跨檔兩輪
- 執行驗證輪次：讓 AI 在每個發現旁邊自報信心分數，用於後續分流
