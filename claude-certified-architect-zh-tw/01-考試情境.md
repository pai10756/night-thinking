# 考試情境

考試採用**情境式題目**。每個情境會描述一個真實的生產環境場景，然後圍繞這個場景出一系列問題。

考試時會從以下 6 個情境中**隨機抽 4 個**出題。

---

## 情境 1：客服問題解決代理人

**場景描述：**
你正在用 Claude Agent SDK 打造一個客服機器人。這個機器人負責處理「高度模糊」的客戶請求——退貨、帳單爭議、帳戶問題等。它透過自訂的 MCP 工具來連接你的後端系統：

| 工具名稱 | 功能 |
|---------|------|
| `get_customer` | 查詢客戶資料（驗證客戶身分用） |
| `lookup_order` | 查詢訂單資料 |
| `process_refund` | 處理退款 |
| `escalate_to_human` | 轉接真人客服 |

你的目標是達成 **80% 以上的一次解決率**（客戶第一次聯繫就能解決問題），同時也要知道什麼時候該轉給真人。

> **為什麼叫「高度模糊」？**
> 因為客戶的問題往往不明確。比如客戶說「我的訂單有問題」，可能是想退貨、可能是缺件、也可能是寄錯地址。AI 必須自己判斷該怎麼處理。

**主要考試領域：** 領域 1（代理人架構）、領域 2（工具設計與 MCP）、領域 5（上下文管理與可靠性）

## 情境 2：用 Claude Code 產生程式碼

**場景描述：**
你正在用 **Claude Code** 來加速軟體開發。你的團隊用它來：
- 產生程式碼
- 重構（改善既有程式碼的結構，但功能不變）
- 除錯（找出和修復 Bug）
- 寫文件

你需要把 Claude Code 整合進開發流程，包含：
- 自訂斜線命令（Slash Commands）——在聊天介面中用 `/命令名` 就能觸發的快捷指令
- 配置 CLAUDE.md 檔案
- 知道什麼時候該用「計畫模式」vs「直接執行」

> **計畫模式 vs 直接執行：**
> - **計畫模式（Plan Mode）**：像設計師先畫藍圖再蓋房子。適合複雜任務。
> - **直接執行（Direct Execution）**：直接動手做。適合簡單任務，比如改一行程式碼。

**主要考試領域：** 領域 3（Claude Code 配置）、領域 5（上下文管理與可靠性）

## 情境 3：多代理人研究系統

**場景描述：**
你正在用 Claude Agent SDK 打造一個**多代理人研究系統**。架構如下：

```
           ┌──────────┐
           │ 協調代理人 │  ← 總指揮，負責分配任務和整合結果
           │(Coordinator)│
           └─────┬────┘
      ┌──────┬───┴───┬──────┐
      ▼      ▼       ▼      ▼
   搜尋代理人 分析代理人 綜合代理人 報告代理人
  (搜網路)  (分析文件) (整合發現) (產生報告)
```

> **什麼是多代理人系統？**
> 想像一個研究團隊：有人負責找資料、有人負責分析、有人負責寫報告。這裡的「代理人」就是一個個專門做特定任務的 AI，由一個「總指揮」來協調它們。

**主要考試領域：** 領域 1（代理人架構）、領域 2（工具設計與 MCP）、領域 5（上下文管理與可靠性）

## 情境 4：用 Claude 提升開發者生產力

**場景描述：**
你正在用 Claude Agent SDK 打造開發者生產力工具。這個 AI 代理人幫助工程師：
- 探索不熟悉的程式碼庫
- 理解舊系統（Legacy System）
- 產生重複性的程式碼模板
- 自動化重複性工作

它使用內建工具：

| 工具 | 功能 |
|-----|------|
| `Read` | 讀取檔案 |
| `Write` | 寫入檔案 |
| `Bash` | 執行命令列指令 |
| `Grep` | 搜尋檔案內容（在程式碼中搜特定文字） |
| `Glob` | 用檔名模式找檔案（例如找所有 `.tsx` 結尾的檔案） |

同時也整合了 MCP 伺服器。

**主要考試領域：** 領域 2（工具設計與 MCP）、領域 3（Claude Code 配置）、領域 1（代理人架構）

## 情境 5：Claude Code 用於持續整合（CI）

**場景描述：**
你正在把 Claude Code 整合進 **CI/CD 流程**。系統會自動：
- 審查程式碼（Code Review）
- 產生測試案例
- 在 Pull Request 上提供回饋

> **什麼是 Pull Request（PR）？**
> 開發者寫好新功能後，會發一個「Pull Request」給團隊，請大家審查。通過審查後才會合併進主程式碼。就像提交報告給主管批核一樣。

你需要設計出能提供**有用回饋**且**減少誤報**的提示詞。

> **什麼是誤報（False Positive）？**
> AI 說「這裡有問題！」但其實沒有。就像煙霧偵測器在煮飯時誤響一樣。太多誤報會讓人失去信任。

**主要考試領域：** 領域 3（Claude Code 配置）、領域 4（提示詞工程與結構化輸出）

## 情境 6：結構化資料擷取

**場景描述：**
你正在打造一個**結構化資料擷取系統**。它要：
- 從非結構化文件（如 PDF、報告、郵件）中自動提取資訊
- 用 JSON Schema 驗證輸出格式

> **什麼是「非結構化」vs「結構化」？**
> - **非結構化**：像一封寫給你的信，資訊散落在各處，沒有固定格式
> - **結構化**：像一張 Excel 表格，每個欄位有明確定義
> - 這個系統的工作就是把「信」變成「表格」

- 維持高準確度
- 優雅處理邊界情況（Edge Cases，意料之外的特殊狀況）
- 和下游系統整合（把擷取的資料送到下一個處理環節）

**主要考試領域：** 領域 4（提示詞工程與結構化輸出）、領域 5（上下文管理與可靠性）
