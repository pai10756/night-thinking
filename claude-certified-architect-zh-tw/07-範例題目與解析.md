# 範例題目與解析

以下範例題目展示了考試的格式和難度。每題都附有詳細解析。

---

## 情境：客服問題解決代理人

### 題目 1

**情境：** 生產環境的資料顯示，有 12% 的案例中，你的 AI 代理人完全跳過了 `get_customer`（查客戶），直接用客戶口頭說的名字去呼叫 `lookup_order`（查訂單），偶爾導致帳戶辨識錯誤和退款到錯誤的帳戶。最能有效解決這個可靠性問題的做法是什麼？

**選項：**
- A) 加入程式化的先決條件，在 `get_customer` 回傳已驗證的客戶 ID 之前，封鎖 `lookup_order` 和 `process_refund` 的呼叫
- B) 在系統提示詞中加強說明，透過 `get_customer` 驗證客戶身分是任何訂單操作前的必要步驟
- C) 加入 few-shot 範例，展示代理人永遠先呼叫 `get_customer`，即使客戶主動提供了訂單細節
- D) 實作一個路由分類器，分析每個請求後只啟用適合該請求類型的工具子集

**正確答案：A**

> **解析（白話版）：**
>
> 當某個步驟**絕對不能跳過**（像驗證客戶身分才能退款），光靠「在指令中寫『記得先驗證客戶』」是不夠的——AI 有時就是會跳過。
>
> 選項 B 和 C 都是「拜託 AI 記得做」，屬於「大部分時候有效但偶爾會失靈」的做法。當失靈的後果是**把錢退到別人帳戶**，這個風險無法接受。
>
> 選項 A 用程式碼硬性規定「沒有客戶 ID 就不能退款」，就像提款機不輸入密碼就不會吐錢——這是 100% 的保證。
>
> 選項 D 解決的是「給哪些工具」的問題，但真正的問題是「工具使用的順序」。

### 題目 2

**情境：** 生產環境的日誌顯示，代理人在使用者問訂單相關問題時（例如「查一下我的訂單 #12345」），經常呼叫 `get_customer` 而不是 `lookup_order`。兩個工具的描述都很簡短（「取得客戶資訊」/「取得訂單詳情」），且接受類似的輸入格式。改善工具選擇可靠度的最有效第一步是什麼？

**選項：**
- A) 在系統提示詞中加入 5-8 個 few-shot 範例，展示正確的工具選擇模式
- B) 擴充每個工具的描述，加入它能處理的輸入格式、範例查詢、邊界情況，以及何時該用它而非類似工具
- C) 實作一個路由層，在每次回合前解析使用者輸入，根據關鍵字和識別碼模式預先選擇工具
- D) 把兩個工具合併成一個 `lookup_entity` 工具，接受任何識別碼並在內部決定查詢哪個後端

**正確答案：B**

> **解析（白話版）：**
>
> AI 選工具就像你在超市選商品——主要靠包裝上的說明。如果兩罐都只寫「調味料」，你不知道哪罐是鹽、哪罐是糖。
>
> 最省力又最有效的做法就是把「包裝說明」寫清楚（選項 B）。
>
> 選項 A（加範例）：增加了每次對話的消耗，而且沒有解決根本問題——描述本身就不清楚。
> 選項 C（路由層）：過度工程化，為了一個「描述沒寫好」的問題建了一整套系統。
> 選項 D（合併）：作為「第一步」來說太大工程了，先把描述改好就能解決大部分問題。

### 題目 3

**情境：** 你的代理人只達成 55% 的一次解決率，遠低於 80% 的目標。日誌顯示它把簡單的案件升級給真人（例如：有照片證據的標準商品損壞替換），卻嘗試自主處理需要政策例外的複雜案件。最有效的改善做法是？

**選項：**
- A) 在系統提示詞中加入明確的升級標準和 few-shot 範例，展示何時升級、何時自主解決
- B) 讓代理人在每次回應前自報信心分數（1-10），低於門檻就自動轉人工
- C) 部署一個訓練過的分類器模型，在主代理人處理前預測哪些請求需要升級
- D) 實作情緒分析，偵測客戶的挫敗感，負面情緒超過門檻就自動升級

**正確答案：A**

> **解析（白話版）：**
>
> 問題的根本原因是 AI 不知道什麼該自己處理、什麼該轉人工——「決策邊界不清楚」。
>
> 選項 A 直接解決這個問題，是最合比例的第一步。
>
> 選項 B 失敗是因為 AI 的自我信心評估很不準——它在困難案件上反而「信心滿滿」（因為它不知道自己不知道什麼）。
>
> 選項 C 過度工程化——需要標記資料和機器學習基礎設施，而提示詞優化都還沒試過。
>
> 選項 D 搞錯了方向——情緒跟案件複雜度是兩回事。客戶可能對一個簡單問題很生氣（等太久），也可能很平靜地提出一個極複雜的需求。

## 情境：用 Claude Code 產生程式碼

### 題目 4

**情境：** 你想建立一個自訂的 `/review` 斜線命令，執行團隊的標準程式碼審查清單。這個命令應該在所有開發者 clone 或 pull 程式碼庫後都能使用。應該把命令檔案放在哪裡？

**選項：**
- A) 專案程式碼庫中的 `.claude/commands/` 目錄
- B) 每位開發者的家目錄 `~/.claude/commands/`
- C) 專案根目錄的 `CLAUDE.md` 檔案中
- D) `.claude/config.json` 中的 commands 陣列

**正確答案：A**

> **解析（白話版）：**
>
> 要讓所有人都能用 = 必須放在會跟著程式碼庫一起分享的地方 = 專案內的 `.claude/commands/`。
>
> 選項 B 是個人目錄，其他人看不到。
> 選項 C 的 CLAUDE.md 是放「規則和說明」，不是放「命令定義」。
> 選項 D 的配置機制在 Claude Code 中根本不存在。

### 題目 5

**情境：** 你被指派將團隊的單體式應用程式重構為微服務。這涉及數十個檔案的變更，需要做出服務邊界和模組相依性的決策。你應該採取什麼方式？

**選項：**
- A) 進入計畫模式，先探索程式碼庫、理解相依性、設計實作方案，然後再動手改
- B) 直接開始做，邊做邊調整，讓實作過程自然揭示服務邊界
- C) 直接執行模式搭配詳盡的前期指令，詳細規定每個服務該怎麼架構
- D) 先用直接執行模式開始，只在遇到預料之外的複雜度時才切換到計畫模式

**正確答案：A**

> **解析（白話版）：**
>
> 從單體式改成微服務就像把一棟大樓拆成多棟獨立的房子——你不會拿到工程圖就開始拆牆。你會先研究結構、了解哪些牆是承重牆、規劃好再動手。
>
> 選項 B：邊做邊發現問題，代價太高（拆錯了要重建）。
> 選項 C：你還沒看過程式碼就假設知道正確架構——這不現實。
> 選項 D：題目已經說了「涉及數十個檔案和架構決策」，複雜度是已知的，不是「可能出現」的。

### 題目 6

**情境：** 你的程式碼庫中不同區域有不同的程式碼慣例。測試檔案散落在各處（如 `Button.test.tsx` 放在 `Button.tsx` 旁邊）。你希望所有測試無論在哪個目錄，都遵循相同的慣例。最好的做法是？

**選項：**
- A) 在 `.claude/rules/` 建立規則檔案，用 YAML frontmatter 的 glob 模式，根據檔案路徑條件式套用慣例
- B) 把所有慣例集中在根目錄的 CLAUDE.md 中，靠 Claude 自行推斷該套用哪一段
- C) 在 `.claude/skills/` 為每種程式碼類型建立技能
- D) 在每個子目錄放一份 CLAUDE.md，寫上該區域的慣例

**正確答案：A**

> **解析（白話版）：**
>
> 用 `.claude/rules/` 加上 glob 模式（如 `**/*.test.tsx`）就能自動對所有測試檔案套用規則，不管它們在哪個目錄。
>
> 選項 B：靠 AI 自己猜不可靠。
> 選項 C：技能需要手動觸發，不符合「自動」的需求。
> 選項 D：測試檔案散落在幾十個目錄中，你不可能在每個目錄都放一份 CLAUDE.md。

## 情境：多代理人研究系統

### 題目 7

**情境：** 研究主題是「AI 對創意產業的影響」。每個子代理人都成功完成了任務，但最終報告只涵蓋了視覺藝術，完全遺漏了音樂、寫作和電影製作。檢查協調者的日誌，發現它把主題分解成：「AI 在數位藝術創作」「AI 在平面設計」「AI 在攝影」。最可能的根本原因是？

**選項：**
- A) 綜合代理人缺少辨識覆蓋缺口的指令
- B) 協調者的任務分解太狹窄，導致子代理人的任務沒有涵蓋所有相關領域
- C) 搜尋代理人的查詢不夠全面
- D) 分析代理人因過度嚴格的相關性標準過濾掉了非視覺類創意產業的來源

**正確答案：B**

> **解析（白話版）：**
>
> 日誌直接揭示了答案：協調者只分出了三個視覺藝術相關的子任務。子代理人們按照被分配的任務正確執行了——問題出在**分配的任務本身就不完整**。
>
> 選項 A、C、D 都在怪下游的代理人，但它們在各自被分配的範圍內都做得很好。

---

### 題目 8

**情境：** 搜尋子代理人在研究複雜主題時逾時了。你需要設計錯誤資訊如何回傳給協調者。哪種錯誤傳遞方式最能幫助智慧恢復？

**選項：**
- A) 回傳結構化的錯誤上下文給協調者，包含錯誤類型、嘗試的查詢、部分結果和可能的替代方案
- B) 在子代理人內部實作自動重試加指數退避，只在所有重試用完後回傳「搜尋不可用」
- C) 在子代理人內部攔截逾時，回傳標記為「成功」的空結果
- D) 直接把逾時例外傳遞到頂層處理器，終止整個研究流程

**正確答案：A**

> **解析（白話版）：**
>
> 給協調者足夠的資訊，它才能做出聰明的決定——要重試？換方式查？還是用已有的部分結果繼續？
>
> 選項 B：「搜尋不可用」太籠統，協調者無法判斷下一步。
> 選項 C：把失敗偽裝成成功，會導致最終報告缺漏但無人知曉。
> 選項 D：因為一個子代理人失敗就放棄所有工作，太浪費了。

### 題目 9

**情境：** 綜合代理人經常需要驗證事實。目前的做法是回到協調者→協調者呼叫搜尋代理人→結果回到綜合代理人，每次任務多 2-3 個來回，延遲增加 40%。評估顯示 85% 是簡單的事實查核（日期、人名、統計數字），15% 需要深入調查。最有效的做法是？

**選項：**
- A) 給綜合代理人一個範圍受限的 `verify_fact` 工具做簡單查核，複雜驗證仍由搜尋代理人通過協調者處理
- B) 讓綜合代理人累積所有驗證需求，在結束時一次批次送給協調者
- C) 給綜合代理人所有搜尋工具，讓它自己處理任何驗證
- D) 讓搜尋代理人在初次搜尋時主動快取額外的上下文

**正確答案：A**

> **解析（白話版）：**
>
> 這是「最小權限原則」的應用——只給綜合代理人處理 85% 常見情況所需的工具，複雜的 15% 仍走正規流程。
>
> 選項 B：批次處理會造成阻塞——後面的綜合步驟可能依賴前面已驗證的事實。
> 選項 C：給太多工具違反職責分離原則，綜合代理人可能濫用搜尋工具。
> 選項 D：搜尋代理人無法可靠預測綜合代理人會需要驗證什麼。

## 情境：Claude Code 用於持續整合

### 題目 10

**情境：** 你的 CI 腳本執行 `claude "分析這個 PR 的安全問題"` 但工作無限期卡住。日誌顯示 Claude Code 在等待互動式輸入。正確的做法是？

**選項：**
- A) 加上 `-p` 旗標：`claude -p "分析這個 PR 的安全問題"`
- B) 設定環境變數 `CLAUDE_HEADLESS=true`
- C) 從 `/dev/null` 重導向 stdin
- D) 加上 `--batch` 旗標

**正確答案：A**

> **解析（白話版）：**
>
> `-p`（或 `--print`）是 Claude Code 在自動化環境中執行的正式做法。它會處理指令、輸出結果、然後結束，不會等人類輸入。
>
> 選項 B、D 描述的功能根本不存在。選項 C 是 Unix 小技巧但不是正確的解法。

### 題目 11

**情境：** 團隊想降低 API 成本。目前有兩個工作流程都用即時 API：(1) 合併前的阻塞式檢查，(2) 隔夜產生的技術債報告。主管提議兩個都改用 Message Batches API 以節省 50% 成本。你怎麼評估？

**選項：**
- A) 只對技術債報告用批次處理；合併前檢查維持即時呼叫
- B) 兩個都改批次處理，用輪詢檢查完成狀態
- C) 兩個都維持即時呼叫
- D) 兩個都改批次處理，設定超時後回退到即時呼叫

**正確答案：A**

> **解析（白話版）：**
>
> 批次 API 省 50% 但最多要等 24 小時。開發者等著合併程式碼，不可能等 24 小時。但隔夜報告本來就是明天早上才看，24 小時完全可以接受。
>
> 所以：急用的→即時 API，不急的→批次 API。對號入座就好。

### 題目 12

**情境：** 一個 PR 修改了 14 個檔案。單次審查所有檔案產生不一致的結果：有些檔案詳細分析、有些只是草草帶過、明顯 Bug 被漏掉、甚至對不同檔案中相同的程式碼模式給出相反的評價。你該怎麼重新架構審查？

**選項：**
- A) 拆成多輪：先逐檔分析各自的問題，再跑一輪跨檔的資料流整合分析
- B) 要求開發者把大 PR 拆成 3-4 個檔案的小 PR
- C) 換更高級的模型，用更大的上下文視窗
- D) 跑三次獨立審查，只標記至少出現兩次的問題

**正確答案：A**

> **解析（白話版）：**
>
> 問題的根本原因是「注意力稀釋」——一次看太多檔案，每個都看不仔細。拆成逐檔 + 跨檔兩輪直接解決這個問題。
>
> 選項 B：把負擔推給開發者，沒改善系統本身。
> 選項 C：更大的上下文視窗不等於更好的注意力品質。
> 選項 D：「多數決」反而會壓制那些只被偶爾發現的真正問題。
