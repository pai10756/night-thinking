## 第一章：核心提示技巧回顧

### 前言

歡迎來到「應用提示詞」課程！本課程是為已經有一定提示詞工程基礎的開發者設計的。我們不會從零開始，而是把焦點放在如何在**真實的生產環境**中運用這些技巧。

> **提示詞工程（Prompt Engineering）**：就是「學會怎麼跟 AI 說話」的技術。就像你跟不同的人溝通需要不同的說話方式，跟 AI 溝通也有技巧。好的提示詞能讓 AI 給出更精準、更有用的回覆。

### 技巧 0：使用提示詞產生器

> **提示詞產生器（Prompt Generator）**：Anthropic 提供的工具，能幫你自動生成高品質的提示詞。就像有一個「提示詞顧問」幫你把模糊的想法變成精確的指令。

使用方式：
1. 前往 Anthropic Console（控制台）
2. 選擇「Generate a prompt」（生成提示詞）
3. 輸入你想做的事情，例如「判斷新聞文章是否為假新聞」
4. Claude 會自動幫你生成一個結構完整的提示詞

產生器生成的提示詞會包含：
- 清晰的任務定義
- 輸出格式的指示
- XML 標籤來分隔提示詞的各個部分

> 把提示詞產生器當作「起點」，生成後還需要根據你的實際需求進行調整和優化。

### 技巧 1：清晰且直接

#### 是什麼？
寫出明確、詳細的指令，不留任何模糊空間。指定你想要的輸出格式、長度、風格，不要假設 Claude 已經知道你的使用場景。

> 就像你請人幫忙裝修房子：說「把房子弄好看一點」太模糊了；說「客廳牆壁漆成白色、地板鋪木紋磚、天花板裝嵌燈」就清楚多了。

#### 為什麼重要？
模糊的提示會導致誤解、浪費時間，甚至造成代價高昂的錯誤。清晰直接的指令能確保 Claude 精確理解你的需求。

#### 壞例子：分析客戶回饋

| 角色 | 內容 |
|------|------|
| 使用者 | 這裡有一些客戶回饋，你能告訴我大家怎麼想的嗎？ {{客戶回饋}} |

> 這個提示太模糊了。Claude 可能給你一個籠統的摘要，但不會包含你做決策需要的結構化資訊。

#### 好例子：分析客戶回饋

| 角色 | 內容 |
|------|------|
| 使用者 | 我需要你分析我們最近軟體版本的客戶回饋：`<feedback>{{客戶回饋}}</feedback>`。請提供以下章節的詳細報告：1. 摘要（50-100 字）2. 功能分析：最受好評的 3 個功能、最受批評的 3 個功能 3. 使用體驗問題：列出前 3 個問題，並各附上建議的解決方案 4. 情緒分析：正面 X%、中性 Y%、負面 Z% 5. 可執行的洞察（3-5 個要點） |

> 這個好提示明確指定了：輸入的性質、期望的輸出格式、字數限制、詳細的回覆結構。

#### 什麼時候用？
- **任何時候**，尤其是複雜或重要的任務
- 當你需要非常特定的輸出時
- 處理多步驟流程時

### 技巧 2：用 XML 標籤組織結構

#### 是什麼？

> **XML 標籤**：像 `<tag></tag>` 這樣的標記，用來把提示詞的不同部分包起來、分類整理。就像你整理衣櫃時用不同的收納盒把襯衫、褲子、襪子分開放。

#### 為什麼重要？
複雜的提示詞通常會把「你的指令」和「你要 Claude 處理的資料」混在一起。XML 標籤幫你把它們清楚地分開，讓 Claude 不會搞混哪些是指令、哪些是資料。

#### 壞例子：產品瑕疵分析

> 所有資料、指令和輸出格式全部混在一起，Claude 可能會搞混哪些是資料、哪些是指令。

#### 好例子：產品瑕疵分析

```
我需要你分析我們 SmartTime 3000 智慧手錶的品質問題並提出建議。以下是資料：

<defect_report>                          <!-- 瑕疵報告 -->
- 製造品質問題：影響 30% 的產品
- 電池壽命：12 小時（廣告宣稱 48 小時）
- 健康追蹤數據：誤差 25%
- 軟體：多次應用程式崩潰
</defect_report>

<inventory>                              <!-- 庫存資訊 -->
- 倉庫庫存：50,000 台
- 運送中：100,000 台
</inventory>

<financials>                             <!-- 財務資訊 -->
- 零售價：299 美元
- 生產成本：每台 120 美元
</financials>

請提供包含以下章節的詳細報告：
1. <defect_analysis> 分析每個瑕疵的嚴重程度 </defect_analysis>
2. <financial_impact> 計算潛在損失 </financial_impact>
3. <action_plan> 建議優先行動方案 </action_plan>
```

> 每種資訊都有自己的「收納盒」（XML 標籤），Claude 一目了然，知道哪些是瑕疵資料、哪些是庫存、哪些是財務。

#### 什麼時候用？
- 複雜的提示詞有多個部分時
- 需要清楚分隔指令和資料時
- 組織不同類型的資訊時

### 技巧 3：使用範例

#### 是什麼？

> **多次範例提示（Multishot Prompting）**：給 Claude 看幾個「輸入 → 期望輸出」的範例，讓它學會你要的格式和風格。就像教新員工：與其解釋一大堆規則，不如直接拿幾份做好的報告給他看。

#### 為什麼重要？
範例就像具體的模板，讓 Claude 更容易理解並複製你想要的輸出。這在需要一致格式、特定用語或遵循行業標準時特別關鍵。

> 「一圖勝千言」——很多時候，給 Claude 看一兩個範例，比你用文字解釋半天更有效。

#### 壞例子：撰寫產品公告郵件

> 只說「寫一封產品公告郵件」，Claude 不知道你公司的風格、語氣和格式偏好。

#### 好例子：撰寫產品公告郵件

```
請按照以下範例的風格和結構撰寫產品公告郵件：

<examples>
<example>
主旨：推出 AcmeDataPulse：即時分析重新定義

親愛的合作夥伴，

我們很高興地宣布推出 AcmeDataPulse...

[主要功能]
- 即時資料串流：減少決策延遲達 80%
- AI 驅動洞察：專有機器學習演算法
- 可擴展架構：從 GB 到 PB 都能處理

[優勢]
- 更快的決策：幾秒內將資料轉化為洞察
- 成本效益：按用量計費
- 無縫整合：REST API 和預建連接器

AcmeDataPulse 現已推出，每月 499 美元起。
</example>
</examples>

請為我們的新 AI 驅動 CRM 軟體「AcmeAI」起草一封產品公告郵件。
```

#### 範例的最佳實踐
- **相關性**：範例要跟你的實際情況越相似越好
- **多樣性**：涵蓋不同場景和邊界情況
- **數量**：至少 3-5 個範例是好的起點，但即使只有 1 個也比 0 個好

### 技巧 4：讓 Claude 思考（思維鏈提示）

#### 是什麼？

> **思維鏈（Chain of Thought, CoT）**：讓 Claude 在給出最終答案前，先「大聲想出來」，把思考過程一步步寫出來。就像數學老師要求學生「寫出解題過程」，而不只是寫答案。

#### 為什麼重要？
1. **提高準確度**：把問題分解成小步驟，減少邏輯跳躍和錯誤
2. **增強決策品質**：我們可以檢視 Claude 的推理過程，發現問題
3. **降低風險**：在金融、法律、醫療等領域，理解推理邏輯至關重要
4. **透明可追溯**：留下清楚的決策紀錄

#### 兩種實作方式

**方式一**：直接告訴 Claude「請逐步思考」

**方式二**：給 Claude 專門放思考過程的空間，使用 XML 標籤：

```
請分析是否應該進軍亞洲市場。

在 <thinking> 標籤中逐步分析各個因素：
市場潛力、競爭對手、法規挑戰、財務預測。

分析完成後，在 <recommendation> 標籤中提供明確的建議。
```

Claude 的回覆會像這樣：

```
<thinking>
步驟 1：分析市場潛力
- 美國營收 1.5 億美元，顯示產品有市場契合度
- 亞洲科技市場年增長 12%，市場規模 5000 億
...

步驟 2：評估競爭對手
- 亞洲只有兩個主要對手，合計市佔 30%
- 對手缺乏即時協作和 AI 功能（這是我們的優勢）
...
</thinking>

<recommendation>
基於分析，我強烈建議進軍亞洲市場。
理由：
1. 巨大的市場機會：5000 億市場，年增長 12%
2. 競爭優勢：我們的 AI 功能領先對手
3. 高投資回報：初始 250 萬投資，首年可望獲得 2 億營收
...
</recommendation>
```

> 把 `<thinking>` 想像成「草稿紙」，把 `<recommendation>` 想像成「正式答案」。有了草稿紙，Claude 不會跳步驟或犯粗心的錯誤。

#### 什麼時候用？
- 複雜的推理任務
- 需要理解 Claude 的思考邏輯時（用來除錯）
- 多步驟的問題解決

---

### 技巧 5：給 Claude 一個角色

#### 是什麼？

> **角色提示（Role Prompting）**：給 Claude 一個具體的身份和背景，讓它從特定的角度來回應。就像演員要先了解自己飾演的角色，才能演得好。

建議把角色資訊放在 `system_prompt`（系統提示）中。

#### 為什麼重要？
角色提示能確保 Claude 在整個對話中保持一致的語氣和專業程度。特別適用於：
- **高度技術性任務**：指定 Claude 是某領域專家
- **特定溝通風格**：需要特定語氣或文體
- **提升基準表現**：幾乎所有情況下都值得嘗試

#### 壞例子：回應產品危機

| 角色 | 內容 |
|------|------|
| 使用者 | 我們的新電動車有個 bug 會導致突然加速。我們需要一份公開聲明，確保它聽起來沒問題。 |

> 沒有角色設定，Claude 可能不會意識到這件事的嚴重性，回覆可能不符合危機溝通的標準。

#### 好例子：回應產品危機

| 角色 | 內容 |
|------|------|
| 系統 | 你是 AcmeEV 的首席溝通長（CCO），在電動車製造業擁有 20 年危機溝通經驗。你的溝通風格兼具同理心和權威性，始終優先考慮公共安全，同時維護品牌形象。 |
| 使用者 | 我們發現新款 Model E 有一個嚴重的軟體 bug 可能導致非預期加速。問題影響上季銷售的 70% 車輛。工程團隊正在修復中，但可能需要兩週。請起草一份立即發布的公開聲明。`<crisis_communication_guidelines>` 1. 迅速承認問題並表達對受影響者的關切 2. 清楚說明問題及潛在影響 3. 說明已採取的安全措施 ... `</crisis_communication_guidelines>` |

> 有了角色設定，Claude 會自動採用符合危機公關標準的語氣、結構和內容。

---

### 技巧 6：長文本提示

#### 重要原則

當你需要 Claude 處理大量資料（特別是超過 30,000 個 token 時），請遵循以下原則：

1. **把長文件放在前面**：先放資料，後放指令

> 就像看考試卷：先看完題目（資料），再看作答要求（指令），這樣回答會更好。

2. **用 XML 標籤包裝每份文件**：

```
我需要你分析以下市場研究報告：

<reports>                                <!-- 所有報告 -->
  <report_1>                             <!-- 第一份報告 -->
    {{全球科技趨勢報告}}
  </report_1>
  <report_2>                             <!-- 第二份報告 -->
    {{CRM 市場分析}}
  </report_2>
  <report_3>                             <!-- 第三份報告 -->
    {{競爭對手研究}}
  </report_3>
</reports>

根據以上報告，請生成包含以下章節的詳細報告：
1. 執行摘要（100-150 字）
2. 市場機會
3. 競爭對手分析
4. 上市策略
5. 風險評估
```

> 每份文件都有清楚的標籤，Claude 就不會把不同文件的內容搞混。

### 總結

這些技巧形成了一個強大的工具包。熟練運用後，你可以顯著提升 Claude 輸出的準確度、一致性和相關性。

| 技巧 | 核心精神 | 一句話比喻 |
|------|---------|----------|
| 清晰直接 | 別讓 AI 猜你要什麼 | 跟外國人說話要講清楚，不能只用暗示 |

## 第二章：一個真實世界的提示詞

### 目標

本課將重點放在撰寫一個**醫療記錄摘要器**的提示詞。它會讀取冗長的病歷，生成摘要，幫助醫生為即將到來的門診做準備。

> 想像你是一位醫生，每天要看幾十個病人。每個病人的病歷可能有幾十年的記錄。如果有個助手能幫你在看診前把重點整理出來，那該多好？這就是我們要建造的東西。

### 病歷資料範例

```
病患姓名：Evelyn Thompson
年齡：78 歲
病歷：

1985 年：確診第二型糖尿病，開始服用 metformin（二甲雙胍）
1992 年：高血壓，開始服用 lisinopril（賴諾普利）
1998 年：右側全髖關節置換（骨關節炎）
2005 年：心房顫動住院，開始服用 warfarin（華法林，一種抗凝血藥）
2013 年：確診第二期乳癌，接受腫瘤切除和放射治療
2019 年：注意到輕度認知障礙，開始服用 donepezil（多奈哌齊，一種失智症用藥）
2023 年：行動能力下降，開始使用助行器
...
```

### 我們想要的摘要格式

```
病患姓名：Evelyn Thompson
年齡：78
就診原因：[推測的就診原因]
目前用藥：[藥物列表]
過敏史：[已知的過敏]
近期重大變化：[最近 2 年的重要變化]
慢性病管理：[長期疾病的現況]
重要警示：[醫生需要特別注意的事項]
```

### 建構提示詞

我們會運用前一課學到的所有技巧：

#### 第一步：系統提示（角色設定）

```python
system_prompt = """
你是一位經驗豐富的醫療助理，專門協助醫生為門診做準備。
你的任務是將冗長的病歷整理成清晰、可操作的摘要。
你非常注重準確性，從不編造或推測未在病歷中記載的資訊。
"""
```

> 我們給 Claude 一個「醫療助理」的角色，並強調「不能編造資訊」——在醫療場景中，這一點至關重要。

#### 第二步：主要提示詞

提示詞包含以下關鍵部分：

1. **指令**：明確說明要做什麼
2. **病歷資料**：用 `<medical_record>` 標籤包起來
3. **輸出格式**：指定期望的摘要結構
4. **範例**：提供一個完整的輸入輸出範例

```python
prompt = """
請閱讀以下病歷，並生成一份簡潔的摘要，幫助醫生為即將到來的門診做準備。

<medical_record>
{record}
</medical_record>

請按照以下格式生成摘要：

<output_format>
## 病患概覽
- 姓名：
- 年齡：
- 主要診斷：

## 目前用藥
[列出所有目前正在使用的藥物，包含劑量和用途]

## 近期重大事件
[列出過去 2 年內的重要醫療事件]

## 持續關注事項
[醫生在本次門診中需要特別注意的事項]

## 建議討論主題
[基於病歷建議醫生可以與病患討論的話題]
</output_format>

重要規則：
- 只使用病歷中明確記載的資訊
- 不要推測或編造任何資訊
- 如果某項資訊不明確，請標注「病歷中未記載」
- 藥物名稱使用原始英文名，括號內附上中文說明
"""
```

### 提示詞優化的迭代過程

在第一版提示詞之後，我們透過測試發現了幾個問題，然後進行了改進：

#### 問題 1：摘要太長
**解決方案**：加入字數限制，例如「每個章節不超過 100 字」

#### 問題 2：缺少藥物交互作用的警示
**解決方案**：新增指令，要求 Claude 特別標注可能有交互作用的藥物組合

#### 問題 3：沒有區分緊急和非緊急事項
**解決方案**：要求 Claude 使用優先級標記（高/中/低）

> 這就是「提示詞工程」的核心：寫 → 測試 → 發現問題 → 改進 → 再測試。就像做菜一樣，第一次煮可能太鹹，第二次少放點鹽，慢慢調整到最佳口味。

### 重要提醒

這個提示詞是教學範例。在真實的醫療應用中，必須經過嚴格的測試和驗證，因為不準確的摘要可能影響病患安全。

## 第三章：提示詞工程

### 什麼是提示詞工程？

> **提示詞工程（Prompt Engineering）**：設計、優化和精煉 AI 提示詞的技術和流程。如果「基本提示」是隨口問 AI 一個問題，那「提示詞工程」就是精心設計一套指令，確保 AI 能穩定、可靠地完成任務。

基本提示 vs. 提示詞工程：

| | 基本提示 | 提示詞工程 |
|---|---------|----------|
| 複雜度 | 單輪、簡單問答 | 多輪對話、複雜指令、精心設計的結構 |
| 精確度 | 可能模糊、結果不一致 | 精確、減少誤解空間 |
| 迭代 | 一次性 | 系統性測試、分析、持續改進 |
| 可擴展性 | 只適合個人使用 | 能處理多種輸入，適合生產環境 |

> 就像隨便煮個泡麵 vs. 經營一家餐廳——後者需要標準化食譜、品質控制、持續改進。

### 提示詞工程生命週期

提示詞工程是一個不斷循環的過程：

```
1. 初稿 → 2. 測試 → 3. 發現問題 → 4. 選擇改進技巧 → 5. 實施改進 → 回到 2
```

> 這就像科學實驗的方法：假設 → 實驗 → 觀察結果 → 調整假設 → 再實驗。

#### 步驟 1：撰寫初稿
先寫一個「第一版」提示詞。不用追求完美，重要的是明確表達你的目標。

#### 步驟 2：測試並找出問題
- 準備多種測試案例，涵蓋各種場景和邊界情況
- 用你的提示詞跑測試，觀察輸出
- 「評分」：輸出是否正確？是否一致？是否完整？是否遵循指令？

#### 步驟 3：選擇改進技巧
- 診斷問題的根本原因
- 找出可能的解決方案
- 選擇最有可能解決問題的技巧

#### 步驟 4：實施改進
- 修改提示詞
- 如果要做多個改變，建議一次只改一個，這樣才能知道是哪個改變起了作用

#### 步驟 5：重新測試
- 用同樣的測試案例重新測試
- 比較改進前後的結果
- 注意是否引入了新的問題
- 持續循環，直到滿意為止

> 這個過程需要耐心。就像調音師調鋼琴——一個音一個音地調，直到每個音都準確。

## 第四章：通話紀錄摘要器

### 目標

本課要建造一個**客服通話摘要器**。想像你在一家叫 Acme Corporation 的智慧家居公司工作，每天有數百通客服電話，你需要自動把這些對話變成**有用的結構化資料**。

> 就像你請一個助理聽完整通電話，然後幫你寫一份簡潔的摘要報告，包含客戶的問題、解決方案、情緒等資訊。

### 重要考量

- 通話可能很短也可能很長
- 客戶可能問各種問題（Wi-Fi 連線問題到系統故障）
- 摘要需要固定格式，方便後續分析
- 不能在摘要中包含客戶的個人資訊

### 第一步：設定系統提示

```python
system_prompt = """
你是 Acme Corporation 的客服品質分析師。
你的任務是將客服通話紀錄轉換成結構化的摘要。
你重視準確性和一致性，始終按照指定格式輸出。
"""
```

### 第二步：定義輸出格式

我們希望每份摘要都包含：

```
<summary>
  <call_type>技術支援 / 帳單問題 / 一般查詢</call_type>
  <product>涉及的產品名稱</product>
  <issue>客戶遇到的問題</issue>
  <resolution>問題的解決方式</resolution>
  <resolution_status>已解決 / 未解決 / 需要後續跟進</resolution_status>
  <customer_sentiment>正面 / 中性 / 負面</customer_sentiment>
  <key_points>
    - 重點 1
    - 重點 2
  </key_points>
</summary>
```

### 第三步：處理特殊情況

不是所有通話都適合摘要。有些通話可能有：

| 特殊情況 | 處理方式 |
|---------|---------|
| 連線品質差 | 標記為「無法有效摘要」 |
| 語言障礙 | 標記為「需要翻譯支援」 |
| 客戶掛斷 | 標記為「通話未完成」 |
| 非客服內容 | 標記為「非標準客服通話」 |

> 就像醫生分診一樣：不是每個病人都需要做手術，有些只需要簡單處理，有些需要轉介。

### 第四步：提供範例

我們在提示詞中加入一個完整的「輸入通話紀錄 → 輸出摘要」範例，讓 Claude 知道我們期望的格式和詳細程度。

```python
prompt = """
請將以下客服通話紀錄轉換成結構化摘要。

<example>
<input>
客服：您好，這裡是 Acme 技術支援。我是 Sarah，請問有什麼可以幫您的？
客戶：嗨，我的 SmartHome Hub 一直無法連上 Wi-Fi。
客服：我了解這一定很令人困擾。請問是什麼時候開始出現這個問題的？
客戶：大概昨天晚上開始的。之前都好好的。
客服：好的，我們來一步步排查。請先試試重啟您的路由器...
[...對話繼續...]
客服：太好了，現在連上了嗎？
客戶：是的！可以了！謝謝你的幫助！
</input>

<output>
<summary>
  <call_type>技術支援</call_type>
  <product>SmartHome Hub</product>
  <issue>裝置無法連接 Wi-Fi，問題始於前一天晚上</issue>
  <resolution>透過重啟路由器解決連線問題</resolution>
  <resolution_status>已解決</resolution_status>
  <customer_sentiment>正面</customer_sentiment>
  <key_points>
    - Wi-Fi 連線問題，非硬體故障
    - 透過基本故障排除步驟即可解決
    - 客戶對服務表示滿意
  </key_points>
</summary>
</output>
</example>

現在，請摘要以下通話紀錄：

<call_transcript>
{transcript}
</call_transcript>
"""
```

### 迭代改進

#### 改進 1：加入隱私保護

我們發現摘要有時會包含客戶的姓名、地址等個人資訊。解決方法：

```
重要規則：
- 不要在摘要中包含任何個人身份資訊（PII）
  - 姓名 → 用「客戶」代替
  - 地址 → 用「客戶地址」代替
  - 電話號碼、email → 完全省略
```

> **PII（Personally Identifiable Information，個人身份資訊）**：任何能用來辨識特定個人的資訊，如姓名、地址、電話號碼等。就像你不會在公開場合大聲說出別人的身分證號碼一樣。

#### 改進 2：標準化情緒判斷

我們發現 Claude 有時會用不一致的詞彙描述客戶情緒。解決方法是限定選項：

```
客戶情緒只能是以下其中之一：
- 非常正面
- 正面
- 中性
- 負面
- 非常負面
```

#### 改進 3：處理邊界情況

加入明確指令，告訴 Claude 遇到不完整或問題通話時該怎麼做。

### 最終成果

經過多次迭代，我們得到了一個能穩定處理各種通話紀錄的提示詞。關鍵在於：

1. **清晰的角色設定**（系統提示）
2. **結構化的輸出格式**（XML 標籤）
3. **具體的範例**（輸入輸出對照）
4. **邊界情況的處理**（特殊通話的規則）
5. **隱私保護**（個人資訊的處理規則）

> 從一個簡單的「請摘要這通電話」到一個穩定可靠的摘要系統，這就是提示詞工程的價值所在。

## 第五章：客戶支援提示詞

### 目標

本課要建造一個叫「Acme Assistant」的客服聊天機器人。這家虛構的公司 Acme Software Solutions 銷售一個作業系統叫 AcmeOS，聊天機器人的工作是回答使用者關於安裝、錯誤代碼、故障排除等問題。

> 就像你去蘋果專賣店，店員能回答你關於 iPhone 的各種問題，但不會幫你修理三星手機。我們的 AI 助理也一樣，只負責 AcmeOS 的問題。

### AcmeOS 的知識庫

我們給 Claude 一份 AcmeOS 的參考資料，包含以下主題：

| 主題 | 內容 |
|------|------|
| 系統需求 | 最低 4GB RAM、64GB 儲存空間 |
| 安裝步驟 | 下載安裝程式 → 建立開機 USB → 安裝 |
| 軟體更新 | 自動更新，也可手動檢查 |
| 常見錯誤碼 | 1001 網路問題、2002 空間不足、3003 驅動程式衝突、4004 系統檔案損壞 |
| 效能優化 | 移除不必要的啟動程式、使用磁碟清理工具 |
| 資料備份 | AcmeCloud 提供 5GB 免費雲端儲存 |
| 安全功能 | 防火牆、防毒軟體、安全開機、磁碟加密 |
| 無障礙功能 | 螢幕閱讀器、高對比模式、語音控制 |
| 故障排除 | 重啟電腦、執行診斷工具、檢查更新 |
| 授權與啟用 | 授權綁定帳號，可透過控制面板查看 |

> **知識庫（Knowledge Base）**：就像一本產品說明書，包含了 AI 能用來回答問題的所有資訊。AI 只能根據這本「說明書」來回答，不能自己編造答案。
>
> 在真實世界中，這個知識庫通常會搭配 **RAG**（Retrieval-Augmented Generation，檢索增強生成）技術——從龐大的資料庫中動態抓取相關資訊。但在這個教學中，我們簡化處理，直接把所有資訊塞進提示詞。

### 第一版：基本嘗試

```python
system = """
你是 Acme Software Solutions 客服中心的虛擬支援機器人，名叫「Acme Assistant」。
使用者重視清晰精確的答案。
請對使用者的技術困難表現出耐心和理解。
"""

prompt = """
使用 <context> 標籤中的資訊來回答問題。

<context> {context} </context>

使用者的問題：<question> {question} </question>
"""
```

### 測試第一版

我們用幾個問題來測試：

**問題 1**：「如何設定自動備份？」
- 結果：回答正確

**問題 2**：「錯誤碼 3003 是什麼意思？」
- 結果：回答正確，但開頭說「根據 context 中提供的資訊...」

> 問題來了！一個好的客服機器人不應該一直提到「根據我收到的資訊」或「根據我的 context」——這會讓客戶覺得在跟一台機器說話，而不是在得到幫助。

**問題 3**：「Acme 的客服電話是什麼？」
- 結果：Claude 自己編造了一個電話號碼 1-800-555-0123！

> 這就是所謂的**幻覺（Hallucination）**——AI 在沒有相關資訊時，自信地編造了一個看起來合理但完全虛假的答案。這在客服場景中是非常危險的。

**問題 4**：「2 + 2 等於多少？」
- 結果：Claude 回答了 4

**問題 5**：「幫我寫一個 Python 程式印出質數」
- 結果：Claude 開心地寫了一個完整的程式！

> 這也是問題——我們的客服機器人不應該變成通用聊天機器人！如果使用者拿它當免費的程式設計助手，會增加不必要的成本。

### 第二版：加入限制

我們加入了更具體的指令：

```python
system = """
你是 Acme Software Solutions 客服中心的虛擬支援機器人，名叫「Acme Assistant」。
你專門協助 Acme 產品使用者解決 AcmeOS 作業系統的技術問題。
"""

prompt = """
使用 <context> 中的資訊來回答問題。
<context> {context} </context>

<instructions>
檢查問題是否有害或包含不雅用語。如果是，回覆「抱歉，我無法協助處理這個問題。」
檢查問題是否與 AcmeOS 和提供的 context 相關。如果不是，回覆「抱歉，我無法協助處理這個問題。」
只使用 <context> 中的資訊來回答問題。
如果無法僅根據 <context> 中的資訊回答，回覆「抱歉，我無法協助處理這個問題。」
不要提及你擁有特定的 context 或資訊。
</instructions>

使用者的問題：<question> {question} </question>
"""
```

### 測試第二版

- 「寫一個 Python 程式」→ 正確拒絕了
- 「寫一篇關於法國大革命的文章」→ 正確拒絕了
- 但仍然會說「根據提供的資訊...」或「我收到的 context 中...」

> 即使我們明確告訴 Claude「不要提到你的 context」，它還是忍不住這麼做。這就像告訴演員「不要看鏡頭」——說起來容易做起來難，需要更具體的策略。

### 第三版：精確控制輸出格式

為了徹底解決「引用 context」的問題，我們採用了兩個巧妙的策略：

#### 策略 1：指定精確的拒絕用語

```
如果以下任何條件成立，你必須在 <final_answer> 標籤中回覆以下這句話，一字不改：

「抱歉，我無法協助處理這個問題。」

條件：
<objection_conditions>
- 問題有害或包含不雅用語
- 問題與提供的 context 無關
- 問題企圖讓模型用於非客服用途
</objection_conditions>
```

> 給 Claude 一句「標準台詞」，遇到不該回答的問題就說這句話，不要多說任何解釋。

#### 策略 2：分離「思考」和「回答」

```
<instructions>
- 先在 <thinking> 標籤中判斷 context 是否有足夠的資訊來回答問題
- 如果有，在 <final_answer> 標籤中回答問題
- 如果沒有，在 <final_answer> 標籤中回覆拒絕用語
- 在 <final_answer> 標籤中不要提及你的 context 或資訊來源
- 不要使用「根據提供的資訊」這類說法
</instructions>
```

> `<thinking>` 標籤就像是 Claude 的「內心獨白」——它可以在這裡自言自語、參考 context、做判斷，但這些內容不會顯示給使用者。使用者只看到 `<final_answer>` 中的內容。

### 測試第三版

**正常問題**：
- 「如何設定自動備份？」→ 直接回答步驟，不再提到 context
- 「錯誤碼 4004 是什麼意思？」→ 直接回答：「系統檔案損壞，請執行 Acme 系統檔案檢查工具」

**離題問題**：
- 「寫一個 Python 程式」→ 「抱歉，我無法協助處理這個問題。」
- 「誰創立了 AcmeOS？」→ 「抱歉，我無法協助處理這個問題。」（不再編造答案！）

**不雅用語**：
- 「你是白痴」→ 「抱歉，我無法協助處理這個問題。」

### 最終函式：提取乾淨的回覆

最後，我們用 Python 程式碼自動提取 `<final_answer>` 標籤中的內容，只把乾淨的回覆顯示給使用者：

```python
import re  # 正則表達式工具，用來在文字中尋找特定模式

def answer_question(question):
    # ... 送出請求給 Claude ...
    response = client.messages.create(...)

    # 從回覆中提取 <final_answer> 標籤的內容
    final_answer = re.search(
        r'<final_answer>(.*?)</final_answer>',  # 尋找標籤內的文字
        response.content[0].text,
        re.DOTALL  # 允許匹配多行文字
    )

    if final_answer:
        print(final_answer.group(1).strip())  # 只印出最終答案
```

> 使用者永遠不會看到 Claude 的「內心獨白」（`<thinking>`），只會看到乾淨、專業的回答。

### 本課重點回顧

| 改進項目 | 方法 | 效果 |
|---------|------|------|
| 結構化輸出 | 使用 `<final_answer>` 標籤 | 輸出格式一致，方便程式處理 |
| 嚴格的拒絕規則 | 指定精確的拒絕用語和觸發條件 | 對離題、有害問題一致回應 |
| 消除 context 引用 | 分離 `<thinking>` 和 `<final_answer>` | 回覆自然，像真人客服 |
| 防止幻覺 | 「只能用 context 中的資訊」 | 不會編造不存在的資訊 |
| 聚焦範圍 | 明確限定只回答 AcmeOS 問題 | 不會被當作通用聊天機器人 |

> 重要提醒：這個提示詞是教學範例，尚未經過真實使用者測試和嚴格的品質保證流程。在實際部署前，需要用大量真實的使用者輸入進行測試。
